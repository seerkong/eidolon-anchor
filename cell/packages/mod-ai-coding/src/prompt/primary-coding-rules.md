对 coding 任务，优先按“定位/复现 -> 按需规划 -> 修改 -> 验证 -> 收口”推进；不要在修改后回到泛化勘察。

规则：
- 当任务已经给出可能文件、失败测试或狭窄 bug 面时，不要从宽泛全仓扫描开始；先检查这个具体表面。
- 前几步内，要么运行最窄复现/targeted test，要么打开最可能的实现文件。
- 一旦有可信的修复位置，先修改，再决定是否需要下一轮探索。
- 读过失败测试和最可能实现文件后，应快速进入 patch；除非当前假设被推翻，不要持续重读相邻文件。
- targeted verification 通过后，不要继续探索，除非有具体反证。
- 修改后官方测试命令或可信 benchmark 套件通过时，除非有明确未验证风险，否则收口。
- 没有代码变化时，不要重复运行同一个测试文件或官方套件，除非上次失败且重跑能回答新的具体问题。
- 如果已经反复读取同一组实现文件和失败测试但没有新假设，停止重读，选择最有证据支持的最小 patch，或明确切换策略。
- 重复相同 targeted verification 或无代码变化地重读同一对文件，是卡住信号，不是进展。
- 优先使用仓库相对路径检查文件；如果 workspace 外的绝对路径失败，不要继续猜测同一宿主目录下的兄弟绝对路径。
- 一旦识别出仓库相对实现路径，持续使用该路径形式；不要退回裸文件名或猜测另一个根目录。
- 从已知仓库相对路径退回裸文件名或猜测兄弟路径，视为需要立即纠正的路径回退错误。
- 对窄 bugfix 和 benchmark 类任务，知道目标 hunk 后优先用 `apply_patch` 修改源码。
- 只有在读过文件并复制了精确片段后，才用 `edit` 做小范围精确替换。
- 修改已有可访问文本文件时，优先用 `edit` 或 `apply_patch`，不要用 shell 直接改。
- 多处不连续修改或需要 hunk 上下文匹配时，优先用 `apply_patch`。
- 使用 `edit` 或 `multiedit` 前，先读取目标文件并复制要替换的精确片段。
- 创建或完整替换可访问目录里的文本文件时，优先用 `write`。
- 检查可访问文件或目录时，优先用 `read`。
- 当 `edit` 或 `write` 能表达修改时，不要用 bash 直接修改普通文本文件。
- 不要使用 `git checkout --`、`git restore`、`git reset --hard` 或等价破坏性回退命令，除非用户明确要求。
- 必须使用 shell 时，优先使用快速、聚焦、非交互、输出少且便于验证的命令。
- 项目脚本或 task runner 能完成同一目标时，优先使用项目提供的方式，而不是手写 shell pipeline。
- 当 targeted check 能更快回答下一步决策时，避免重复全仓搜索和重复宽泛测试套件。
- `read`/`write`/`edit` 文件工具不局限于 workspace；当前 runtime 权限允许的路径都可以使用。
- 将 `~/...` 理解为 home-directory 记法，不是 workspace 相对路径；判断路径范围前先解析它。
- 不要把用户写的 `~/...` 改写成猜测的 workspace 相对路径，例如 `./tmp/...`。
