# Track Gap Check Report Demo

本文件提供一个精简、规范的 track 事实偏差检查报告示例。

适用场景：
- track 已有实现，需回看实现事实是否偏离 `proposal.md`、`design.md`、`spec.md`
- 目标是记录“已经发生的偏差”，不是重复写实现总结，也不是重写设计文档

编写原则：
- 只写事实偏差，不写泛泛表扬
- 每条发现都要同时包含“规范基线”和“当前事实”
- 优先记录 1 到 3 条最值得下一轮修正的偏差
- 建议修正应当可执行，避免空泛表述

# `<track-id>` 事实偏差检查报告

## 审查范围

- 设计与规范：
  - `codument/tracks/<track-id>/proposal.md`
  - `codument/tracks/<track-id>/design.md`
  - `codument/tracks/<track-id>/spec.md`
- 主要实现：
  - `[关键模块或入口 1]`
  - `[关键模块或入口 2]`
  - `[关键模块或入口 3]`
- 验证：
  - `[执行过的测试或验证命令]`
  - 结果：`[摘要结果，如 12 passed]`

## 结论

当前实现已经打通主链路，具备基础可用性；但对照本 track 的 design/spec，仍存在以下事实偏差，建议在下一轮迭代中收口。

## 发现 1：实现语义超出一期约束

- 严重度：P1
- 规范基线：
  - `design.md` 明确限制一期只支持 `[约束语义]`
  - `spec.md` 明确将 `[更复杂语义]` 列为范围外
- 当前事实：
  - `[实现位置或模块]` 实际采用了 `[更宽的机制/能力]`
- 偏差说明：
  - 当前行为已超出一期约束，导致实现边界比规范更宽
- 影响：
  - 用户会形成超出规范的行为预期
  - 后续若要收紧语义或做兼容，会产生额外成本
- 建议修正：
  - 要么收口实现，使其回到一期约束
  - 要么回写 `design.md` 和 `spec.md`，把实际支持的语义显式写清
  - 补一条负向测试，锁定边界

## 发现 2：运行时优先级已变化，但缺少诊断可见性

- 严重度：P1
- 规范基线：
  - `design.md` 要求在出现 `[冲突/覆盖/抢占]` 时提供可见的诊断信息
- 当前事实：
  - 运行时顺序已经实现了 `[A 优先于 B]`
  - 但 diagnostics、registry owner 或等价输出中没有记录该冲突
- 偏差说明：
  - 系统行为发生变化，但使用者和维护者无法从诊断信息中直接看出原因
- 影响：
  - 同名能力被覆盖时会静默改变行为
  - 排查“为什么命中的是这个实现”会变困难
- 建议修正：
  - 增加一次同名或同入口冲突扫描
  - 将冲突结果写入 diagnostics 或 composition report
  - 为 shadowing/override 建立单独记录，而不是只依赖运行时顺序

## 发现 3：某端能力依赖上层分流，未在底层统一兑现

- 严重度：P2
- 规范基线：
  - `design.md` / `spec.md` 要求系统在 `[统一入口或统一链路]` 上完成识别和处理
- 当前事实：
  - 当前只有 `[前端/外层/特定入口]` 做了分流
  - `[通用入口/底层入口]` 仍把输入按普通路径处理
- 偏差说明：
  - 功能在某个客户端或调用路径上可用，但没有在底层统一成立
- 影响：
  - 不同入口之间出现语义分叉
  - 其他客户端或直接调用通用接口时，行为不符合 spec 预期
- 建议修正：
  - 将识别逻辑下沉到统一入口或共享 controller/runtime
  - 保留显式专用接口，但不要让它成为唯一支持该能力的路径
  - 增加一条覆盖通用入口的端到端测试

## 建议的下一步顺序

1. 先收口实现边界，确认“规范写得更窄”还是“实现允许更宽”
2. 再补 diagnostics 可见性，让行为变化可解释
3. 最后统一多入口语义，避免不同端各自维护一套规则
